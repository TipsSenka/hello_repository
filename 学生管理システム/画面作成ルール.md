# 画面作成ルール（学生一覧系）

## 1. 目的
- 同じ設計ルールで、学生管理系の画面を横展開できるようにする。
- データはHTMLに直書きせず、JSONファイルを唯一のデータソースにする。
- 一覧表示、ソート、エラー表示の実装方針を共通化する。

## 2. ファイル構成ルール
- 画面本体: 専科の学生管理.html のようなHTMLファイルを作成する。
- データ: students.json のようなJSONファイルを同階層に配置する。
- 相対パス fetch("students.json") で取得できる構成にする。

## 3. データ定義ルール（JSON）
- 形式は配列（Array）とする。
- 1件のデータはオブジェクトで、最低限以下の3項目を持つ。
  - name: 氏名（文字列）
  - id: 学籍番号（文字列）
  - email: メールアドレス（文字列）
- 画面で使う項目名とJSONキー名は一致させる。

## 4. HTML構造ルール
- 見出し（h1）で画面名を表示する。
- 一覧は table 要素で表示し、thead と tbody を分ける。
- tbody には一意のidを付与する（例: student-list）。
- 列ヘッダーは「表示名」と「ソート対象キー」の対応を明確にする。

## 5. JavaScript実装ルール
- 画面初期化で loadStudents() を1回だけ呼ぶ。
- データ表示は displayStudents(students) に集約する。
- displayStudents 内では、描画前に tbody の中身を必ず空にする。
- データ件数が0件の場合は「データがありません」を1行表示する。
- 取得失敗時は catch でエラー行（colspan）を表示する。

## 6. ソート実装ルール
- sortStudents(property) は1つだけ定義する（重複定義しない）。
- ソートは表示中データまたは保持済みデータを対象に実行する。
- 文字列比較は localeCompare("ja") を使う。
- 同じ列を連続クリックした場合は昇順/降順を切り替える。
- 現在のソート状態はヘッダーで ▲ / ▼ を表示して可視化する。

## 7. 禁止ルール
- HTMLコメント内の名簿を正データとして運用しない。
- ソートのたびに不要な二重fetchをしない。
- 描画関数の外でDOMへ直接行追加しない（描画責務を分離する）。
- 同名関数を複数回定義しない。

## 8. 再利用手順（他画面作成時）
1. 画面用HTMLを新規作成する。
2. 同階層に対象データJSONを作成する。
3. display関数、load関数、sort関数をテンプレートとして流用する。
4. 列定義（見出し名とキー名）だけ画面要件に合わせて差し替える。
5. 0件表示・エラー表示・ソート切替・▲▼表示を動作確認する。

## 9. 品質チェック項目
- 初回表示でデータが全件表示される。
- 各列のソートが動作する。
- 同じ列の再クリックで昇順/降順が反転する。
- ソート後に行が重複しない。
- JSON読込失敗時に画面が崩れずメッセージを表示する。
